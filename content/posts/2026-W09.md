## [ ] CLAUDE.md files are a mess

**Platform:** Both
**Source:** 2026-02-claude-md-files-are-a-mess.md

**Twitter (thread):**

1/ CLAUDE.md files are a mess. Here's why I stopped using them.

2/ Every file has the same name. You end up with a dozen CLAUDE.md files across your repo. The only way to know what's in each one is to open it.

A file called testing-conventions.md tells you what's inside. CLAUDE.md tells you nothing.

3/ They load every turn, whether relevant or not. The ETH Zurich study found this is exactly what hurts. Agents read testing conventions and then apply them when you asked for a quick utility function.

Irrelevant instructions compete for attention and steer the model wrong.

4/ What I use instead: a skills/ directory with files named for what they do. Each one loads only when that skill is invoked, not passively injected into every conversation.

One set of docs, symlinked to every AI tool I use. Agent-agnostic by default.

**LinkedIn:**
I stopped using CLAUDE.md files a few months ago. The concept is solid but the implementation creates problems that compound over time.

The naming problem is the most obvious. Every file is called CLAUDE.md. You end up with a dozen of them across your repo and no way to tell from the filename which one covers testing vs. deployment vs. database patterns. A file called testing-conventions.md tells you exactly what's in it.

They sprawl. New rules get added to whichever CLAUDE.md someone happens to be near. Standards contradict each other across files. Nobody can answer "what rules is my agent following right now?" because the answer is scattered across fifteen files in twelve directories.

On teams it gets worse. Whoever writes the CLAUDE.md first wins. Their preferences get committed and imposed on everyone. Six months later the file is a mix of actual project standards, personal preferences, and stale instructions for a tech stack you've migrated off of.

What I use instead: a directory of named markdown files, each loaded when the relevant skill is invoked. Not passively injected into every conversation. Agent-agnostic. Auditable. And you can actually see everything in one place.

## [ ] The ETH Zurich vs Vercel debate on CLAUDE.md

**Platform:** Both
**Source:** 2026-02-claude-md-files-are-a-mess.md

**Twitter:**
ETH Zurich: CLAUDE.md files reduce task success by 3% and increase costs 20%.
Vercel: Our AGENTS.md hit 100% pass rate.

They're both right. They're just measuring different things.

Vercel's file contained API docs for new APIs the model had never seen. That's injecting missing knowledge. ETH Zurich tested style preferences the model could figure out on its own.

Lesson: use these files for knowledge your agent can't discover, not preferences it can.

**LinkedIn:**
There's an interesting tension in the CLAUDE.md debate right now.

ETH Zurich published a study showing CLAUDE.md and AGENTS.md files reduce task success rates by about 3% while increasing inference costs by over 20%. Meanwhile, Vercel published data showing their AGENTS.md hit a 100% pass rate where skills alone only managed 53%.

Both findings are real. They're measuring fundamentally different things.

Vercel's AGENTS.md contained compressed API documentation for new APIs the model had literally never seen before. That's injecting knowledge the model doesn't have. Of course it helped.

The ETH Zurich study tested files full of style preferences, coding conventions, and project structure. The kind of information a model can figure out on its own by reading the codebase. Loading all of that every turn made agents too obedient. They applied testing conventions when asked to write a simple utility function. The irrelevant instructions competed for attention and steered the model wrong.

The takeaway: use persistent context files for knowledge your agent genuinely cannot discover on its own. New APIs, internal tooling docs, domain-specific workflows. Don't use them for preferences and conventions the model can infer from the code around it.
