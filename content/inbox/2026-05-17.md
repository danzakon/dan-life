## 20260517-RS-001 — RBAC, AI Agents, and the Modern Foundry: Why Centralized Access Control Is the Wedge

**Source:** Research report
**Raw file:** `research/reports/20260517-rbac-ai-agents-modern-foundry-wedge.md`
**Status:** raw

### Executive Summary

End-to-end analysis of access control fundamentals (RBAC, ABAC, ReBAC, PBAC, Zanzibar), the enterprise IAM incumbent landscape, the new agent/MCP identity problem, Palantir Foundry's actual moat, and the case for building an AI-native enterprise operating system with agent RBAC as the entry wedge.

Core thesis: the user's intuition is right — enterprises have no centralized access control plane for AI agents — but the greenfield window is closing fast. Microsoft Entra Agent ID (May 2026) and Okta for AI Agents (April 30, 2026) are both GA. Palo Alto bought CyberArk for $25B in Feb 2026. Cisco is reportedly in talks for Astrix at $250–350M. Pure-play NHI startups will be acqui-hired into security platforms within 12–18 months.

The real opportunity is one layer up: the policy + ontology + operational application plane *built on top of* the access plane. That is what Foundry is. Foundry's moat is the stacked combination (ontology + FDE motion + lineage-aware mandatory access controls + defense-grade compliance), not any single piece. The MCP gateway is the right wedge because it is the session layer for agent-tool interactions and the only structural chokepoint that touches every agent-tool packet. The land sequence: OSS gateway → $250K cloud governance tier (CAIO-approvable) → $5M enterprise platform (CISO + CIO + CAIO triad) → $50M FDE-led ontology builds in one named vertical (BFSI recommended) with one named SI partner (Accenture or Deloitte).

Token spend governance is real ($30–40B in 2026 LLM API spend, Anthropic's March 2026 unbundling forces enterprises to attribute spend) but is the tertiary wedge — build as a feature of the access plane, not a separate product.
